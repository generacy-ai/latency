# Research: ExecutionLease & ClusterRegistration Types

## Technology Decisions

### 1. Schema Location: `src/api/execution/`

**Decision**: Place schemas under `src/api/` (not `src/types/`).

**Rationale**: The codebase distinguishes between:
- `src/api/` — Platform API schemas (auth, subscription, organization): schemas owned by the platform that define external contracts
- `src/types/` — Cross-component communication types (agency-humancy, knowledge-store): schemas that define boundaries between internal components

ExecutionLease and ClusterRegistration are platform-level API contracts consumed by orchestration and billing services, making `src/api/` the correct home.

### 2. ID Strategy: Mixed ULID + Plain Branded Strings

**Decision**: LeaseId and ClusterId use ULID validation; QueueItemId, JobId, ProjectId use plain branded strings.

**Rationale**: LeaseId and ClusterId are internally generated by the platform and can enforce ULID format. QueueItemId, JobId, and ProjectId may originate from Firestore document IDs or external systems that don't use ULID format. Plain branded strings (`z.string().min(1).transform(...)`) provide type safety without imposing format constraints on external systems.

**Alternatives considered**:
- All ULID: Would break if external systems use non-ULID IDs
- All plain strings: Would lose format validation on internally-controlled IDs
- UUID v4: Less sortable than ULID, not the established pattern in this codebase

### 3. No `createdAt`/`updatedAt` Fields

**Decision**: Add only an `id` field beyond the spec's original fields.

**Rationale**: The domain-specific timestamps already serve these roles:
- `grantedAt` / `connectedAt` → creation timestamp
- `lastHeartbeat` / `lastSeen` → last-update timestamp

Adding generic `createdAt`/`updatedAt` alongside them would be confusing and redundant.

### 4. Enum-Only Status Validation

**Decision**: Status fields are validated as enums only. No transition validation helpers.

**Rationale**: Zod schemas validate objects at rest — they can't validate state transitions (no access to previous state). Transition validation (`active → releasing`) is stateful logic that belongs in the consuming service layer, not the schema definition. The schema's job is to ensure the status value is valid, not that the transition is valid.

### 5. Export Path: `./api` Only

**Decision**: Add a single `./api` subpath export. No `./api/execution` subpath.

**Rationale**: A single `./api` path covers all API schemas and is forward-compatible as more API types are added. Per-subdirectory paths (e.g., `./api/auth`, `./api/execution`) are overkill for now and can be added later if needed. Consumers import from `@generacy-ai/latency/api`.

## Implementation Patterns

### Versioned Namespace Pattern (Established)

Every schema in `src/api/` follows this pattern:
```typescript
export namespace SchemaName {
  export const V1 = z.object({ ... });
  export type V1 = z.infer<typeof V1>;
  export const Latest = V1;
  export type Latest = V1;
  export const VERSIONS = { v1: V1 } as const;
  export function getVersion(version: keyof typeof VERSIONS) { return VERSIONS[version]; }
}
export const SchemaNameSchema = SchemaName.Latest;
export type SchemaName = SchemaName.Latest;
export const parseSchemaName = (data: unknown) => SchemaNameSchema.parse(data);
export const safeParseSchemaName = (data: unknown) => SchemaNameSchema.safeParse(data);
```

### Branded Type Pattern (Established)

IDs use branded types for compile-time safety:
```typescript
export type LeaseId = string & { readonly __brand: 'LeaseId' };
export const LeaseIdSchema = z.string()
  .regex(ULID_REGEX, 'Invalid ULID format for LeaseId')
  .transform((val) => val as LeaseId);
```

### Cross-Field Validation (Established)

Uses `.refine()` with custom error paths:
```typescript
.refine(
  (data) => new Date(data.lastHeartbeat) >= new Date(data.grantedAt),
  { message: '...', path: ['lastHeartbeat'] }
)
```
