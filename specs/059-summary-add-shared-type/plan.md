## Summary

Add V2 schemas to `ExecutionLease` and `ClusterRegistration` namespaces with `userId` (Firebase Auth UID) and `orgId` (ULID) fields to support per-user concurrency enforcement. Introduce a new `UserId` branded type in `common/ids.ts` using `.min(1)` string validation (not ULID, since Firebase Auth UIDs are opaque strings). Follow the established V1→V2 versioning pattern from `GeneracySubscriptionTier`.

## Implementation Steps

### Step 1: Add UserId branded type (`common/ids.ts`)

Add `UserId` branded type and `UserIdSchema` following the pattern of `QueueItemId` / `JobId` (non-ULID, `.min(1)` validation):

```typescript
export type UserId = string & { readonly __brand: 'UserId' };
export const UserIdSchema = z
  .string()
  .min(1, 'UserId must not be empty')
  .transform((val) => val as UserId);
```

No `generateUserId()` function — Firebase Auth UIDs come from Firebase, not generated locally.

### Step 2: Export UserId from common (`common/index.ts`)

Add `UserId` and `UserIdSchema` to the existing ID exports block.

### Step 3: Add V2 to ExecutionLease (`execution-lease.ts`)

Follow the `GeneracySubscriptionTier` V1→V2 pattern:

1. Extract V1 field definitions into a `v1Shape` const (no behavior change)
2. Define `V2` schema extending `v1Shape` with `userId` (UserIdSchema) and `orgId` (OrganizationIdSchema)
3. Repeat the `lastHeartbeat >= grantedAt` refinement on V2
4. Update `Latest` → V2
5. Add `v2` to `VERSIONS` registry

Import additions: `UserIdSchema` and `OrganizationIdSchema`.

### Step 4: Add V2 to ClusterRegistration (`cluster-registration.ts`)

Same pattern as Step 3:

1. Extract V1 field definitions into a `v1Shape` const
2. Define `V2` extending with `userId` and `orgId`
3. Repeat both refinements (`busy + idle <= total`, `lastSeen >= connectedAt`)
4. Update `Latest` → V2, add `v2` to `VERSIONS`

### Step 5: Update execution index exports (`api/execution/index.ts`)

Add `UserId` type re-export from common for consumer convenience:

```typescript
export type { LeaseId, ClusterId, QueueItemId, JobId, ProjectId, UserId } from '../../common/ids.js';
```

### Step 6: Add V2 unit tests

For both `execution-lease.test.ts` and `cluster-registration.test.ts`:

- Valid V2 objects with userId + orgId
- Rejection of missing userId / orgId
- Rejection of empty userId
- Rejection of invalid orgId (non-ULID)
- V1 schemas still work via `getVersion('v1')` (backward compat)
- `Latest` now points to V2
- `getVersion('v2')` works
- Parse helpers use V2 (Latest)

## Key Technical Decisions

| Decision | Rationale |
|----------|-----------|
| `UserId` uses `.min(1)` not ULID regex | Firebase Auth UIDs are opaque strings, not ULIDs |
| No `generateUserId()` | UIDs come from Firebase Auth, not generated locally |
| V2 fields are **required** (not optional) | Spec explicitly states this is an acceptable breaking change for enforcement correctness |
| Extract `v1Shape` const | Avoids duplicating field definitions between V1 and V2 (follows `generacy-tier.ts` `baseShape` pattern) |
| Repeat refinements on V2 | Each version must be self-contained for independent validation (matches `generacy-tier.ts` pattern) |
